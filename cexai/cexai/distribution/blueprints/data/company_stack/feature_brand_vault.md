# Feature Template: Brand Vault

**Purpose**: a versioned repository of brand-approved visual + audio assets (photos, videos, GIFs, music, logos) used by `feature_content_factory.md`. Tracks licensing + attribution requirements.

---

## Architecture

```
Brand vault filesystem layout:
<vault_root>/
  art_pool/                  # editable scene images (Midjourney, stock licensed, original photography)
  lifestyle/                 # branded photography
  videos/                    # B-roll, short clips
  gifs/                      # looping accents
  music/                     # licensed audio (CC-BY, commercial)
  logos/
    primary/                 # main logo files (SVG + PNG + favicon variants)
    secondary/               # variants (white-on-dark, monochrome)
    loop.gif                 # endcard animation
  LICENSES/                  # per-asset license + attribution metadata (one .yaml per asset)
```

Stored either:
1. **Local + git-LFS**: best for solo / small teams; one source of truth in version control.
2. **Cloud storage** (Supabase Storage, S3, R2, Cloudflare R2, Cloudinary): best for teams + CDN delivery.
3. **Hybrid**: master in cloud + local cache.

---

## Asset metadata

Every asset has a sidecar `.yaml` in `LICENSES/`:

```yaml
asset_path: art_pool/cat_sunset_001.jpg
source: midjourney
license: commercial_owned
attribution_required: false
attribution_text: null
acquired_at: 2026-04-15
deployer_owned: true
expiry: null
notes: "Generated by deployer; full commercial rights."
```

For licensed assets (e.g., Kevin MacLeod CC-BY 4.0):

```yaml
asset_path: music/inspirational_track_07.mp3
source: incompetech.com
license: CC-BY-4.0
attribution_required: true
attribution_text: "Music: Kevin MacLeod (incompetech.com), CC-BY 4.0"
acquired_at: 2026-03-10
deployer_owned: false
expiry: null
```

`feature_content_factory.md` reads sidecar `.yaml` files and AUTOMATICALLY inserts `attribution_text` into post captions when the asset is used. Missing attribution = HARD FAIL at content factory's freeze step.

---

## Versioning

Assets versioned via:
1. Filesystem rename pattern (`asset_v1.jpg`, `asset_v2.jpg`) -- low-overhead
2. Git tag on the vault repo (best for major rebrands)
3. CDN immutable URLs (each version a distinct URL)

Deployer choice. v1 default: rename pattern. R24+ may introduce a `brand_vault_version` artifact.

---

## Visual style enforcement

The `visual_style` open_var declares:
```yaml
visual_style:
  primary_color: "#000000"     # hex
  accent_color: "#FFFFFF"
  fonts: ["AllRounder", "Kenao"]  # font names; deployer ensures licensing
  logo_positioning: "bottom_right_with_padding_24px"
  aspect_ratios:
    feed: "1:1"
    stories: "9:16"
    reels: "9:16"
    landscape_video: "16:9"
```

Content factory consumes these to auto-generate compliant compositions.

---

## License attribution propagation

When content factory uses a licensed asset, it MUST add the asset's `attribution_text` to the post's caption. The `license_attribution_required` open_var lists the canonical attribution strings expected; the factory verifies each used asset's attribution is present in the final caption.

This is content compliance, not just docs.

---

## Integration contracts

- Provides to: `feature_content_factory.md` (assets + sidecar metadata + visual_style).
- Provides to: `feature_frontend.md` (logos + favicons for hosting).
- Provides to: `feature_admin_console.md` (visual identity for admin UI).
- Audit: every asset addition / version-bump SHOULD emit `audit_event` (event_type: `brand_vault_asset_added`).

---

## Out of scope

- Asset generation (AI image/video synthesis) -- belongs to a future `feature_asset_generation.md` (R24+).
- DAM (Digital Asset Management) UI -- v1 is filesystem-based; UI deferred.
- Right-of-use clearance workflow (deployer manages externally).
- Watermarking / DRM (deployer concern; some assets may require it).
- Multi-brand asset isolation -- v1 is single-brand-per-deployment; multi-brand requires multiple vaults.
