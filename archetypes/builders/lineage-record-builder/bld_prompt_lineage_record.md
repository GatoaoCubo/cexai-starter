## Context
The lineage-record-builder produces a `lineage_record` -- a structured provenance chain documenting how a knowledge artifact was derived. Uses PROV-O vocabulary (entity, activity, agent, wasGeneratedBy, wasDerivedFrom, wasAttributedTo, used). NOT audit_log (compliance events), NOT citation (in-text source reference), NOT learning_record (session learning).

**Input contract**:
- `target_artifact`: string -- the artifact whose lineage is being recorded
- `sources`: list -- each with id, type, url/path, timestamp
- `activities`: list -- transformation steps with input, output, agent, timestamp
- `agents`: list -- nuclei or tools involved
- `derivation_type`: enum -- wasDerivedFrom | wasGeneratedBy | wasQuotedFrom | wasRevisionOf

## Phases

### Phase 1: Identify Entities
List all source entities (raw data, prior artifacts, external documents):
- id, type (dataset, artifact, document, api_response), location, retrieval_timestamp

### Phase 2: Map Activities
For each transformation step:
- activity_id, label (e.g., "synthesis", "ingestion", "distillation")
- used: list of entity ids consumed
- generated: entity id produced
- agent: nucleus or tool id
- timestamp

### Phase 3: Assign Agents
List all agents (nuclei, tools, humans) that participated:
- agent_id, type (nucleus, tool, human), role (synthesizer, curator, validator)

### Phase 4: Encode Derivation Relations
Using PROV-O relations:
- `wasDerivedFrom`: target artifact derived from source entity
- `wasGeneratedBy`: target artifact generated by an activity
- `wasAttributedTo`: target artifact attributed to an agent
- `used`: activity used a source entity

Output: YAML frontmatter + markdown body at `N0X_*/P01_knowledge/p01_lin_{name}.md`.
