# Domain Knowledge: prompt_package

## Executive Summary

A `prompt_package` is the frozen, self-contained handoff artifact produced when 8F runs in Mode
B (decomposed): a reasoning-capable model (Opus/Sonnet, Stage 1) runs F1-F4 once and writes the
result to disk; a cheap generation-only model (Haiku/Flash/Ollama, Stage 2) reads that ONE file
and executes F6 PRODUCE only, with no live tools and no memory of Stage 1's reasoning. Tools then
run F7-F8 (Stage 3: doctor + compile + commit + signal). This is Mode B's entire mechanism made
concrete as a typed artifact -- `.claude/rules/8f-reasoning.md`'s "Mode B (decomposed)" row.

## Patterns

| Pattern | Application |
|---------|-------------|
| 2 real production paths | (1) auto-generated by `cex_8f_runner.py` Stage 1 during `Task tool: dispatch decompose`; (2) hand-authored via this builder or the template, for examples/tests |
| Pre-resolved context | Never embed a live MCP call or retrieval query -- resolve the result FIRST, embed the text |
| Template embedding | The TEMPLATE section is the target kind's own `bld_output` ISO, frontmatter-stripped |
| Stage-2 model hint | `stage_2_model_hint` is a SOFT preference; `cex_decompose.py::_resolve_models` precedence is CLI flag > env var > YAML tier > this hint > default |
| Fallback chain | `cex_decompose.py::stage_2` retries `tiers.decompose.stage_2_fallback` models on Stage-2 failure |
| Wikilink gate (W2) | Stage 3 gates the freshly-produced ARTIFACT (not the package) for fabricated `links` before F8 -- `cex_decompose.py` lines 223-288 |
| DGUARD (factual-synthesis guard) | Flag-gated (`CEX_DECOMPOSE_GUARD`/`--guard-on-factual`); warns/upgrades/refuses when a factual kind (`knowledge_card` et al.) is about to be decomposed -- `cex_decompose.py` lines 463-594 |
| A3 quality-floor escalation | Flag-gated (`CEX_DECOMPOSE_ESCALATE`); re-runs Stage 2 at the next escalation-ladder tier while the artifact scores below `tiers.decompose.min_score` (default 8.0) |

## Anti-Patterns

| Anti-Pattern | Why it fails |
|-------------|-------------|
| Assuming `p03_pp_{{task_id}}.md` is the file you will find on disk | Real instances (`.cex/runtime/packages/`, `stress_decompose_packages/`) use `pp_{kind}_{id}.md` with NO `p03_` prefix -- a genuine, unreconciled registry-vs-code gap |
| Embedding a live tool call in `## CONTEXT` | Stage 2 has no live tools; an embedded call just sits there unexecuted |
| Letting Stage 2 run F1-F4 | Defeats the entire purpose of decomposition -- the cheap model MUST only run F6 |
| Using Mode B for factual-synthesis kinds without the DGUARD warning | MISSION_BENCH measured decompose (topology C) at q=1.2 for T1 knowledge_card BOTH rounds -- 8.4x worse than the frontier path (D=17.75/E=19.60) |
| Treating `[FILL: ...]` as the only valid marker | The interface doc names it, but the real template + real writer both use `{{PLACEHOLDER}}` Mustache syntax -- accept both, do not silently pick one |
| Skipping the Stage-3 wikilink gate | Bench2 proved 3/3 rail-governed cheap producers fabricated `links` -- ungated artifacts reach F8 with hallucinated references |

## Application

1. Confirm F1-F4 are already resolved for the target kind (do not run them yourself as the packager)
2. Harvest constraints/identity/context/plan into the 4 canonical body sections
3. Embed the target kind's `bld_output` ISO as `## TEMPLATE`
4. Write to `.cex/runtime/packages/pp_{target_kind}_{id}.md` (the REAL convention)
5. Validate against `p06_if_prompt_package.md`'s Validation Rules before handoff
6. Let Stage 3 tools run F7-F8 -- never self-score, never self-commit
