# Domain Knowledge: trace_config
## Executive Summary
Trace configs define the observability contract for agent execution — specifying what to capture, at what rate, where to export, and how long to retain. Modern LLM agent tracing extends distributed tracing concepts (OpenTelemetry spans) with agent-specific concerns: token accounting, prompt privacy, 8F pipeline stage mapping, and quality regression detection. They differ from quality gates (artifact scoring), log configs (message formatting), and metric configs (counter definitions).

## Patterns
- **Span hierarchy for LLM agents**: extend OpenTelemetry with agent-specific spans
| Level | Span | Key Attributes | Purpose |
|-------|------|---------------|---------|
| Root | `cex.pipeline` | nucleus, kind, intent | One per build/request |
| L1 | `cex.8f.{function}` | function (F1-F8), duration_ms | Pipeline stage tracking |
| L2 | `cex.llm.call` | model, tokens_in, tokens_out | LLM invocation cost/latency |
| L2 | `cex.tool.call` | tool_name, status | External tool invocation |
| L3 | `cex.memory.read` | memory_type, source | Memory/knowledge retrieval |
- **Token accounting**: every LLM span must track token usage
| Attribute | Type | Purpose |
|-----------|------|---------|
| cex.tokens.input | int | Tokens sent to model |
| cex.tokens.output | int | Tokens generated by model |
| cex.tokens.total | int | Sum — cost calculation |
| cex.tokens.cached | int | Tokens served from cache (cost reduction) |
- **Error classification**: categorize errors for automated response
| Category | Error Types | Action |
|----------|------------|--------|
| transient | rate_limit, timeout, connection_reset | Auto-retry with backoff |
| permanent | auth_failure, model_not_found, invalid_schema | Escalate to operator |
| degraded | fallback_triggered, quality_below_threshold | Continue with reduced capability |
- **Selective capture**: privacy-first approach to content capture
| Environment | Prompts | Responses | Metadata | Rationale |
|-------------|---------|-----------|----------|-----------|
| development | YES | YES | YES | Full visibility for debugging |
| staging | NO | NO | YES | Privacy-safe, metadata sufficient |
| production | NO | NO | YES | Privacy-first, cost-effective |
| evaluation | YES | YES | YES | Quality analysis requires full content |
| Source | Concept | Application |
|--------|---------|-------------|
| OpenTelemetry | Distributed traces with spans | Root → child span hierarchy |
| LangSmith | LLM-specific tracing | Prompt/response capture for eval |
| Jaeger | Trace visualization | Search by trace ID, latency distribution |
| Honeycomb | High-cardinality analysis | Query by any span attribute |
| W3C Trace Context | Trace propagation | Cross-service trace correlation |
## Anti-Patterns
| Anti-Pattern | Why it fails |
|-------------|-------------|
| Capturing all prompts in production | PII exposure, 95% of storage is never queried |
| No sampling in production | 8% latency overhead, storage explosion |
| No retention policy | Trace data grows unbounded — 200GB in months |
| No error classification | All errors treated equally — slower incident response |
| No 8F span mapping | Traces show execution but not which pipeline stage failed |
| Flat spans (no hierarchy) | Cannot distinguish pipeline stages from LLM calls from tool calls |
| No token accounting | Cannot track cost per request or detect token budget anomalies |
## Application
1. Determine environment (dev/staging/prod/eval) — this sets sample_rate and capture defaults
2. Select exporter based on infrastructure (console for local, OTLP for prod, LangSmith for eval)
3. Define span hierarchy following the 5-level pattern (root, 8F, LLM, tool, memory)
4. Set capture rules: metadata always, content only in dev/eval
5. Configure token accounting in every LLM span
6. Classify errors into transient/permanent/degraded
7. Define 3-tier retention: hot (7d), warm (30d), cold (90d)
8. Reference endpoint credentials via env vars, never embed
